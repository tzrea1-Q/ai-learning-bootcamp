## 4. 遇到的问题与阻碍
### 4.1 今日主要问题
1. `rewrite` 虽然能返回结果，但第一次真实样例回归暴露出书面化和轻度扩写问题，说明“可用”和“稳定”之间还有明显差距。
2. 统一错误响应格式时，需要同时保留 FastAPI 默认 `422` 校验结构，又要让 `500 / 502` 的返回稳定统一，异常边界稍不收紧就会变得混乱。
3. PostgreSQL Docker 骨架的配置检查通过了，但本机 Docker daemon 未启动，导致 `docker compose up -d postgres` 无法真正把容器拉起。

### 4.2 问题原因分析
- 技术原因：LLM 输出风格天然不稳定；requests 异常和上游结构异常需要主动做统一映射；Docker 骨架验证依赖本机 daemon 状态，而不是只看 compose 文件本身。
- 认知原因：如果只盯着“接口能跑”和“测试通过”，很容易忽略输出风格、错误结构和环境链路这种更接近交付质量的问题。
- 执行原因：今天实际做的是收口和留痕，文档、验证、复盘和路径整理占用了不少时间，因此虽然任务全部完成，但还留下了一批未提交的收尾改动。

### 4.3 当前解决情况
- 已解决：`rewrite` 的第一轮风格漂移已做收紧；`500 / 502` 已统一错误结构；文本输入已加最大长度 `4000` 约束；最小 GitHub Actions workflow 已建好；Docker 骨架的配置一致性已确认。
- 未解决：`rewrite` 仍有轻微书面化倾向；`422` 还没有纳入统一错误层；GitHub Actions 尚未在 GitHub 上实际跑通；Docker 容器还未在本机成功启动。
- 需要明天继续处理的：整理并提交当前未提交改动；决定是否统一 `422`；如果 Docker 环境可用，再补一次真正的 PostgreSQL 启动与 `pg_isready` 验证。

## 5. 今日学习收获
### 5.1 新理解的概念
- “错误响应统一”不是把异常都包一层这么简单，而是要明确哪些状态码先统一、哪些保留框架默认行为，以及错误码、错误摘要、定位信息各自承担什么职责。
- 输入边界也是接口契约的一部分；像“最大长度 4000”这种约束，只有同时落到代码、测试、文档和实际返回里，才算真正生效。

### 5.2 新掌握的方法
- 通过 `TestClient` 快速验证输入边界和 `422` 实际返回，可以在不依赖网络和外部服务的情况下收紧接口契约。
- 用 `docs/traces/` 集中管理测试、联调、回归和样例留痕，再用 `07-postmortems/` 沉淀项目级复盘，可以把“原始证据”和“总结复盘”分层管理。

### 5.3 今天最重要的一个认知升级
- 工程化收口的关键不在于再多做一个接口，而在于把“输出质量、错误结构、输入边界、验证留痕、CI 骨架、环境一致性”这些容易被忽略的基础件一起补齐。
