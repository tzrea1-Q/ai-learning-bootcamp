## 4. 遇到的问题与阻碍
### 4.1 今日主要问题
1. MiniMax 虽然请求成功，但真实响应里混入了 `<think>...</think>` 片段，说明“接口能通”不等于“结果可直接交付”。
2. `POST /rewrite` 的结果可用但风格不够稳定，存在扩写过度、自动加标题或输出过长的问题。
3. 当前完成的是本地闭环和最小日志，统一错误格式、提交记录和 CI 还没有收口。

### 4.2 问题原因分析
- 技术原因：LLM 输出天然存在结构和风格不稳定性；初版接口对上游响应做了网络层验证，但没有提前把输出清洁度和格式稳定性纳入强约束。
- 认知原因：早期更关注“能不能调通”，对“是否能作为稳定 API 输出”重视不够，直到真实返回 `<think>` 片段才暴露问题。
- 执行原因：今天主要目标是把第 1 周最小闭环做通，因此优先完成了接口、测试、日志、README 和复盘，暂未继续推进 CI 和统一错误响应格式。

### 4.3 当前解决情况
- 已解决：MiniMax 连通性、三类文本接口、`<think>` 清洗、最小日志输出、本地真实调用验证、测试留痕、失败案例沉淀。
- 未解决：`rewrite` 输出风格稳定性仍不够；统一错误响应格式和更完整日志策略未完成；当前还没有 commit 和 CI 留痕。
- 需要明天继续处理的：先收紧 `rewrite` prompt 和验收标准，再补统一错误格式或接口响应规范。

## 5. 今日学习收获
### 5.1 新理解的概念
- LLM 接口的“成功”至少分两层：网络调用成功和业务输出可交付成功；HTTP 200 只是第一层。
- 文本处理 API 不只是包一层模型调用，还必须考虑 prompt、响应清洗、错误映射、日志和测试。

### 5.2 新掌握的方法
- 用 `monkeypatch` 替换外部 API 调用，先把接口行为测稳定，再做真实联调。
- 先跑最小脚本验证上游，再用 `uvicorn + Invoke-RestMethod` 跑完整接口链路，最后把异常现象沉淀成 postmortem。

### 5.3 今天最重要的一个认知升级
- AI 应用开发的关键不是“模型能回答”，而是把模型的不稳定输出包进一个可验证、可排查、可复盘的工程壳里。
