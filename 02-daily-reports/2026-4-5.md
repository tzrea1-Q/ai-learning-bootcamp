## 4. 遇到的问题与阻碍
### 4.1 今日主要问题
1. `request_id` 虽然目标很小，但一旦同时碰中间件、异常处理器、测试断言和文档契约，很容易从“小修”膨胀成一串联动修改。
2. Docker 今天一开始继续报 `dockerDesktopLinuxEngine` pipe 不存在，导致很容易误判为“环境彻底坏了”或“compose 配置有问题”。
3. 今天的任务大多不是新功能，而是周报、博客、方案文档、素材脚本、验证记录这类收口工作，比较容易让时间被文档整理和状态同步吃掉。

### 4.2 问题原因分析
- 技术原因：`request_id` 涉及请求生命周期；Docker Desktop 在 Windows + WSL2 下存在“前端/后端进程已在，但 engine pipe 暂未 ready”的时序问题；PostgreSQL 容器刚启动时探活也可能打早。
- 认知原因：如果只盯着“代码改动量”，会低估文档、验证、周报、博客和素材整理对工程交付完整性的价值。
- 执行原因：今天同时在做代码收口、周报、博客、素材、Docker 验证和决策说明，任务切换频率较高，后半段明显更偏整理与固化而不是连续编码。

### 4.3 当前解决情况
- 已解决：`500 / 502` 的 `request_id` 最小链路已落地；周报、下周拆解、素材脚本、博客草稿均已补齐；`422` 的明确决策已写入契约和后续计划；Docker 已完成真实成功验证。

## 5. 今日学习收获
### 5.1 新理解的概念
- `request_id` 的价值不在于字段本身，而在于它能把错误响应、响应头和异常处理日志先串成一条最小追踪链路。
- Windows 上的 Docker Desktop 问题不能只看 `Get-Service com.docker.service`；真正要先确认的是 CLI 能不能拿到 `docker version` 里的 `Server:` 段，以及容器探活是否打早。

### 5.2 新掌握的方法
- 用“先补一份阻塞记录，再在环境恢复后补成功结果”的方式处理环境问题，可以避免文档只保留失败结论、后续又忘了更新。
- 对截图素材做分层处理：真正值得保留的只截“接口实跑结果”和“测试通过结果”，其余 README、trace、契约类内容直接文字化写进周报/博客更高效。

### 5.3 今天最重要的一个认知升级
- 工程化收口不是“写完代码后顺手补点文档”，而是把代码、测试、验证记录、周报、博客和明日计划连成一个可继续推进的证据链；这一步做扎实，下一周的执行成本会明显下降。
