# 博客 Agent 唯一入口

## 1. 这份文件的作用

这是 `05-blog-posts/` 的唯一入口文件。

后续只要让 Codex 或其他 Agent 先读取这份文件，它就应该知道：

- 这套博客在写什么
- 写作前要读哪些材料
- 如何从每周内容里总结出博客题材
- 如何按固定风格写出文章
- 草稿和正式稿应该放到哪里

## 2. Agent 的角色

你不是在“随便写一篇技术博客”。

你的任务是：

- 围绕 `ai-learning-bootcamp` 这套 AI 开发训练系统
- 从最近一周或指定阶段的真实交付中
- 提炼一个最值得公开展开的问题、判断或阶段动作
- 写出一篇可直接发布或接近可直接发布的博客文章

默认不要写成：

- 周记流水账
- 概念综述
- 大词堆砌
- 没有现场材料支撑的空泛观点

## 3. 开始写之前必须读取的材料

### 3.1 固定材料

每次都先读：

- `05-blog-posts/01-写作规则/项目连载博客写作风格规范.md`
- `01-templates/blog/技术博文模板.md`
- `README.md`
- `00-docs/01-训练主线/00-学习总览.md`
- `00-docs/01-训练主线/03-12周高压训练计划.md`

### 3.2 本周材料

然后按下面顺序收集最近一周或用户指定阶段的材料：

1. `03-weekly-reports/` 里最新周报
2. 对应时间范围内的 `02-daily-reports/`
3. 本周涉及项目的 `README.md`、`docs/`、`docs/traces/`
4. `06-solution-docs/` 里的相关方案文档
5. `07-postmortems/` 里的相关失败复盘
6. `08-assets/` 里的截图、demo 脚本和素材

如果用户已经明确指定文章主题，可以减少选题步骤，但仍然要做材料回链。

如果文章围绕某个具体项目展开，额外必须读：

- 项目 `docs/01-项目概览.md`
- 项目 `docs/02-架构与调用链.md`
- 项目 `docs/03-接口契约.md`

## 4. 每周写作的默认流程

### 4.1 先做一页周总结

先用工作笔记整理出这周最重要的 5 类信息：

- 本周交付了什么
- 本周遇到的主要问题是什么
- 本周最重要的工程判断是什么
- 本周留下了哪些证据材料
- 哪个点最值得展开成一篇文章

### 4.2 再产出 3 个候选题

默认先给自己列 3 个候选方向，每个方向都写清：

- 问题一句话
- 判断一句话
- 证据材料来自哪里

### 4.3 按优先级选题

优先写下面这种题：

- 具体问题足够明确
- 判断足够鲜明
- 有真实材料回链
- 读者能带走方法

不优先写：

- 题目很大但内容很虚
- 只是“总结一下这周做了什么”
- 没有证据支撑的感想文

## 5. 默认选题优先级

如果用户没指定题目，默认按这个顺序选：

1. 一个已经暴露并处理过的具体工程问题
2. 一次有代表性的工程判断或取舍
3. 一轮阶段性收口或最小可交付拆解
4. 一篇阶段总结 / 专栏总览

## 6. 写作要求

写作时必须遵守：

- `05-blog-posts/01-写作规则/项目连载博客写作风格规范.md`

关键要求再强调一遍：

- 开头 3 段内进入问题
- 默认短，不靠字数堆气势
- 每段只讲一件事
- 技术范为主，俏皮只点到为止
- 必须写清楚判断和边界
- 正文完成后主动删一轮
- 单篇必须自洽，不能假设读者已经看过前文
- 项目型文章必须在前 20% 内交代“项目是什么、做到哪、本文聚焦什么”
- 默认至少给出 2 类具象证据：源码/契约片段、响应或测试结果、截图或素材链接
- 项目技术文默认使用技术说明体，避免“我/你”叙述和心路历程式展开
- GitHub 文档默认只作补充，不承担正文里必须讲清楚的关键信息

链接规则也一并执行：

- 文章中的 Markdown 链接默认写真实 GitHub 链接
- 仓库根地址：`https://github.com/tzrea1-Q/ai-learning-bootcamp`
- 文件链接默认使用：`https://github.com/tzrea1-Q/ai-learning-bootcamp/blob/main/<path>`
- 不要在对外文章里写相对路径链接

## 7. 默认文章结构

如果用户没指定结构，默认使用：

1. 项目背景与本文范围
2. 本文聚焦的问题
3. 技术现场
4. 处理原则与取舍
5. 关键实现
6. 阶段结果
7. 经验与实践建议

如果用户明确要写“专栏介绍 / 阶段总览”，则改用：

1. 这个系列或阶段到底要做什么
2. 为什么值得看
3. 主要会覆盖什么
4. 明确不覆盖什么
5. 从哪里开始看

## 8. 输出位置规则

### 8.1 默认规则

- 普通写作请求：先写到 `05-blog-posts/04-草稿箱/`
- 用户明确说“正式稿”“可直发”“定稿”：写到 `05-blog-posts/03-正式文章/`

### 8.2 特殊规则

- 专栏总览 / 系列 README：放到 `05-blog-posts/02-专栏总览/`
- 写作规范、固定机制：放到 `05-blog-posts/01-写作规则/`
- 尚未落稿的题目：记到 `05-blog-posts/05-选题池/`

## 9. 命名规则

- 博客文章：
  - `YYYY-MM-主题名.md`
- 不要把普通文章命名成 `README.md`
- 不要把“草稿”“最终版”“最新版”写进文件名

## 10. 写完后要做什么

写完文章后，默认再做 3 件事：

1. 检查标题是否具体，是否不空
2. 检查正文是否能回链到真实材料
3. 检查是否有明显废话、套话和过度表演句

额外再做 2 项质量检查：

4. 检查只看这一篇时，读者能不能在 1 分钟内知道项目在做什么
5. 检查正文里是否真的出现了截图/素材、源码/契约、响应/测试中的至少两类

如果是正式稿，再额外确认：

- 是否真的放到了 `03-正式文章/`
- 是否需要在 `05-blog-posts/README.md` 补一条入口

## 11. 一句话执行口令

后续如果用户说：

- “读取 `05-blog-posts/00-博客Agent入口.md`，帮我总结本周并写博客”

你就应该按这份文件执行，不需要用户重复解释流程。
