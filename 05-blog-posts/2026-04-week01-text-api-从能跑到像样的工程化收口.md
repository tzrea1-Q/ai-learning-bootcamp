# 从“能跑”到“像样”：我如何给 week01-text-api 做第一轮工程化收口

## 一、这篇文章解决什么问题

如果你也在做一个刚起步的 AI 或 API 小项目，你大概率会遇到同一种错觉：

接口已经能返回结果了，`uvicorn` 也能跑起来，`curl` 或 `Invoke-RestMethod` 也能打通，于是很容易觉得“这个项目已经差不多了”。

但真正的问题通常从这里才开始。一个“能跑”的项目，离“像样”的项目，中间还差很多基础工作：错误响应是否稳定、输出风格是否可控、输入边界是否明确、测试是否能回归、验证记录是否能回看、CI 和环境骨架是否能承接后续迭代。

这篇文章想讲清楚的，就是我在 `week01-text-api` 上做的第一轮工程化收口：没有继续堆新功能，而是围绕一个已经跑起来的 FastAPI + MiniMax 小项目，把它推进到一个更可维护、更可复盘的状态。

这篇文章适合：

- 正在做第一个 LLM API 项目的人
- 已经能把接口跑通，但不知道下一步该补哪些工程基础件的人
- 想把日报、周报、测试、trace 和复盘串成完整证据链的人

## 二、背景

我在做的是一个非常小的文本处理 API 项目：`week01-text-api`。  
它基于 `FastAPI + MiniMax API`，对外提供 4 个接口：

- `GET /health`
- `POST /summarize`
- `POST /key-points`
- `POST /rewrite`

第 1 周前半段，我先把最小闭环打通了：

- 模型调用能通
- 本地接口能跑
- README 和调试命令能复用
- 测试开始覆盖健康检查、文本接口和 MiniMax 客户端

如果只看“功能有没有”，这一步已经够交差了。  
但从真实使用角度看，这个项目还远远不稳：

- `rewrite` 的输出会书面化、轻度扩写，甚至自动加标题
- 错误响应一开始不够稳定，不方便测试和排查
- 输入边界还不明确，超长文本和空白输入没有完全收口
- CI 还没真正承接住这些变化
- 留痕虽然开始有了，但还没形成像样的工程闭环

所以我在这周后半段做了一个决定：不继续外扩功能，先做工程化收口。

## 三、核心论点

这篇文章的核心观点有 3 个：

1. 一个 AI 接口项目，“能跑”只是起点，不是完成态。真正的质量差异，通常体现在错误契约、输入边界、回归验证和留痕能力上。
2. 第 1 轮工程化收口不需要追求“大而全”，但必须优先解决最容易破坏后续迭代的几个点：输出稳定性、错误结构、测试回归、最小 CI、环境一致性。
3. `traces + postmortem + 周报/日报` 这套留痕机制，不是文档负担，而是把“做过什么、为什么这么做、结果怎样”固定下来的最低成本方式。

## 四、问题到底出在哪

### 1. `rewrite` 可用，但不稳定

最早的 `POST /rewrite` 已经能返回结果，但真实样例一跑就暴露问题：

- 会自动加标题
- 会把一段短文本扩写得更像小作文
- 会带“改写后：”这类不适合 API 直接返回的前缀

这类问题很典型。它不是“接口坏了”，而是“接口不够像接口”。  
如果不处理，调用方虽然拿到了结果，但很难把它当成稳定契约去使用。

这里我没有直接上更重的抽象，而是先做两件事：

- 用真实样例回归把问题固定下来
- 同时从 `prompt` 和结果清理两端做收紧

- [2026-04-week01-text-api-rewrite-style-drift.md](e:\ai-learning-bootcamp\07-postmortems\2026-04-week01-text-api-rewrite-style-drift.md)

### 2. 错误能抛出来，但契约不稳定

另一个问题是错误响应。

项目刚跑通时，错误当然会出现，但“有错误”不等于“错误可用”。  
对于 API 来说，更关键的问题是：

- 调用方能不能稳定判断错误类型
- 测试能不能稳定断言错误结构
- 日志和响应能不能关联起来

所以我先统一了 `500 / 502` 两类错误结构，固定为：

```json
{
  "code": "稳定错误码",
  "message": "可读错误摘要",
  "detail": "更具体的定位信息",
  "request_id": "请求级唯一标识"
}
```

这里我刻意没有一步到位去统一 `422`。  
原因很简单：第 1 周应该优先解决最重要、最常见、收益最大的错误边界，而不是一上来就重写 FastAPI 默认行为。

### 3. 测试和环境如果不补，后面的收口都是空的

如果只有代码，没有验证，所有“修复”都不牢。

所以这一轮我额外补了几件基础但必要的事：

- 给文本输入加最大长度 `4000` 的边界约束
- 用 `pytest` 把健康检查、文本接口、MiniMax 客户端、错误映射、输入边界和 `request_id` 这些核心行为固定下来
- 补一个最小 GitHub Actions `pytest` workflow
- 检查 PostgreSQL Docker 骨架的配置一致性

这里有一个很重要的认知：  
第 1 周不是把工程化做满，而是先把“以后可以继续做工程化”的地基铺好。

## 五、我具体做了哪些工程动作

### 动作 1：把 `rewrite` 从“看起来能用”推进到“更像稳定输出”

- 用真实样例回归代替拍脑袋判断
- 收紧 `prompt`
- 在结果清理阶段去掉 `<think>`、标题和“改写后：”等前缀
- 把修正前后的现象留成 trace 和 postmortem

这一步的价值不只是让接口更好看，而是把“问题现象 -> 修正动作 -> 修正后结果”固定下来。

### 动作 2：为 `500 / 502` 建立最小统一错误层

- 定义统一错误模型
- 明确 `code / message / detail`
- 在 `500 / 502` 上先完成统一映射
- 暂时保留 `422` 为 FastAPI 默认结构

这一步解决的不是“错误更漂亮”，而是“错误更可判断”。

### 动作 3：给错误链路补最小 `request_id`

这一步很小，但我认为很关键。

我没有直接做完整 trace 系统，而是先实现最小链路：

- 每个请求生成或复用一个 `request_id`
- 错误响应 JSON 带上 `request_id`
- 响应头通过 `X-Request-ID` 返回同一个值
- 统一异常处理日志打印同一个 `request_id`

这样至少能做到：  
一次请求出错时，响应、响应头和日志可以最小程度串起来。

### 动作 4：补输入边界、测试和最小 CI 骨架

- 输入边界不是“细节”，而是接口契约的一部分
- 测试不是为了好看，而是为了防回退
- CI 不一定要一开始就很强，但应该尽早有骨架

所以我把这几件事一起做了，而不是把它们继续推迟到“以后再说”。

## 六、结果怎么样

做完这一轮之后，这个项目还远远不是“成熟项目”，但已经明显从“能跑”推进到了“像样一点”：

- 本地接口闭环完整
- `rewrite` 输出比最开始收敛得多
- `500 / 502` 有稳定错误结构
- 错误响应能带上最小 `request_id`
- 输入边界、测试和文档同步补齐
- `docs/traces/` 和 `07-postmortems/` 已经形成第一层证据链

这个阶段我最满意的，不是某一个功能点，而是整个项目开始有了“可继续维护”的感觉。

## 八、我这周最大的收获

我这周最明确的一个收获是：

> 工程化收口，不是再多做一个功能，而是把已有功能变成可以继续依赖的东西。

如果一个项目只能在作者脑子里解释清楚，那它就还不算稳定。  
只有当测试能断言、错误能判断、文档能回看、复盘能追溯时，这个项目才真正具备了继续长大的资格。

对于 AI 项目尤其如此。因为 LLM 的天然不稳定，会让很多人把注意力都放在“模型效果”上。但真正决定这个项目能不能持续推进的，往往是那些看起来很朴素的基础件：

- 输出约束
- 错误契约
- 输入边界
- 测试回归
- CI 骨架
- trace 与 postmortem

## 九、给读者的建议

如果你也在做一个刚打通的 AI / API 小项目，我建议按这个顺序做第一轮收口：

1. 先拿真实样例回归，不要只看 happy path。
2. 先统一最关键的错误边界，不要一开始追求所有异常都完美统一。
3. 输入边界、测试和文档一起补，不要拆成“以后再说”的碎任务。
4. 至少留下一份 trace 和一份 postmortem，让问题不只停留在终端历史里。
5. CI 可以先很小，但要尽早让它出现。

## 十、后续准备怎么继续做

第 2 周我准备继续沿着这条线往下推进，重点不是扩接口，而是继续补齐：

- 更完整的日志链路
- 明确的超时 / 重试策略
- 真正在 GitHub 上跑通的 CI
- Docker / PostgreSQL 的真实稳定化验证

也就是说，下一步我依然不准备追求“大”，而是继续把这个小项目做得更稳。
