# 项目连载博客写作风格规范

## 1. 这份文档解决什么问题

这份文档不是“写作参考”，而是后续博客默认执行的硬约束。

目标只有一个：

- 让后续文章在写完后就接近可直接发布，不再靠二次大改来补救语气、篇幅和节奏

适用范围：

- `ai-learning-bootcamp` 整体连载
- 当前示例项目 `text-api`
- 后续项目、工程化收口、失败复盘、阶段总结

## 2. 专栏的目标与边界

这个系列要做的，不是追 AI 热点，也不是写“我今天学了什么”的学习日记。

它要持续回答一件事：

- 一个真实 AI 项目，怎样从“能跑”慢慢做成“像样”

读者读完后，应该拿走 3 类东西：

- 对项目推进顺序的理解
- 对工程判断的参考
- 对公开输出方法的复用路径

不追求的东西：

- 情绪感染力大于信息密度
- 标题热闹、正文发虚
- 看起来很会说，但看完不知道该怎么做

## 3. 风格基调

整个专栏统一使用下面这条基调：

- 技术范为主，略带小俏皮

拆开说就是：

- 技术范：信息密度高，判断清楚，少空话
- 小俏皮：允许有一点轻微的幽默和画面感，但只起调味作用，不能抢正文

更具体一点：

- 可以写“接口能回 `200`，不代表项目已经出院”
- 可以写“这个坑不大，但很会拖时间”
- 不要写“太香了”“绝了”“无敌”“炸裂”
- 不要写过度表演式反问和情绪堆砌

原则：

- 文章先像工程师写的，再像作者写的

## 4. 绝对写作规则

后续所有文章默认遵守下面这些规则。

### 4.1 先上问题，禁止长铺垫

开头前 3 段内，必须出现下面至少两项：

- 具体问题
- 具体场景
- 这篇文章的核心判断

不要一开头就写：

- 行业背景大综述
- “最近这几年……”
- “很多人都知道……”
- 大段价值宣言

### 4.2 默认短，不靠字数制造厚重感

默认目标：

- 单篇正文尽量控制在 `1200-2500` 字

可以更长，但必须满足至少一个条件：

- 有明确的排查过程需要保留
- 有必要保留多轮取舍
- 有足够多的真实证据支撑

如果一篇文章超过 `3000` 字，落稿前必须主动删一轮。

### 4.3 每段只做一件事

每段尽量只承载一个信息点。

避免：

- 一段里同时讲背景、判断、例子、感受和结论

推荐：

- 一段只讲一个判断
- 一段只讲一个例子
- 一段只讲一个转折

### 4.4 少形容，多结论

优先写：

- 做了什么
- 为什么做
- 为什么不先做别的
- 结果怎样

少写：

- “很重要”“非常关键”“意义重大”这类无支撑形容词

能直接下结论时，不要先绕三句。

### 4.5 轻微俏皮，不许浮夸

允许：

- 偶尔一句轻松比喻
- 偶尔一句自嘲式小转折
- 偶尔一句带点画面感的判断

不允许：

- 连续抖机灵
- 故作锋利
- 标题和正文都在用力表演
- 为了“有网感”牺牲技术精度

判断标准：

- 把一句俏皮话删掉后，文章依然成立

### 4.6 单篇必须自洽，不能假设读者看过前文

默认假设读者是第一次点进这篇文章。

所以项目型、周更型、问题复盘型文章，必须在正文前 20% 以内补足最小背景，至少说清 3 件事：

- 这是个什么项目
- 当前已经做到哪一步
- 这篇文章只聚焦哪一个问题或动作

不要把这些背景都丢给仓库 README 或上一篇文章。

判断标准：

- 读者只看这一篇，也能知道项目大致在做什么
- 读者不需要先补三篇前文，才能理解本文为什么值得看

## 5. 语气规范

### 5.1 推荐语气

- 冷静
- 直接
- 清醒
- 诚实
- 有判断
- 技术说明体

### 5.2 禁止语气

- 浮夸
- 鸡汤
- 说教
- 焦虑营销
- 自我感动

### 5.3 推荐表达

- “这篇文章只想讲清楚一个问题”
- “对当前这个项目体量来说，先做这个更划算”
- “当前阶段先收这个点，收益更高”
- “这不是最佳实践，只是当前阶段更合适的做法”

### 5.4 不推荐表达

- “这将彻底改变……”
- “你一定要……”
- “如果不会这个就……”
- “我终于悟了……”
- “这个方法无敌……”

### 5.5 叙述视角

项目型博客、工程问题复盘、阶段收口文章，默认使用技术说明体，不使用心路历程体。

优先主语：

- 当前项目
- 当前实现
- 本轮收口
- 服务端 / 调用方 / 上游模型
- 接口 / 测试 / 日志 / 文档

默认避免：

- “我当时……”
- “你会发现……”
- “如果你……”
- “这里我选择……”

例外：

- 专栏开篇、阶段宣言、作者说明类文章，可以保留少量自然的一人称
- 但项目技术文默认不靠“我/你”推进叙述

## 6. 标题规则

标题要满足 3 个要求：

- 具体
- 可读
- 有判断

推荐方向：

- 一个具体问题 + 一个明确判断
- 一个阶段动作 + 一个结果
- 一个误判 + 一次修正

推荐例子：

- 《给错误响应补一个字段还不够：`request_id` 真正解决的是什么问题》
- 《`rewrite` 能跑还不够：我怎么收紧一个文本接口的输出风格》
- 《Docker 不是坏了，是还没 ready：一次本地环境误判排查记录》

不推荐例子：

- 《关于 AI 工程化的一些思考》
- 《我的项目开发心得》
- 《如何做好 AI 项目》

问题是太大、太虚、太像套话。

## 7. 文章结构规则

后续默认沿用这个结构：

1. 项目背景与本文范围
2. 本文聚焦的问题
3. 技术现场或问题暴露方式
4. 处理原则与取舍
5. 关键实现
6. 阶段结果
7. 经验与实践建议

如果是开篇、阶段总结、README 型文章，可以改成：

1. 这个系列或阶段到底要做什么
2. 为什么值得看
3. 主要会覆盖什么
4. 明确不覆盖什么
5. 下一步怎么看

## 8. 证据标准

能回链就回链，能落到仓库材料就不要只靠印象。

优先引用：

- 项目 README
- `docs/traces/`
- 周报 / 日报
- 方案文档
- postmortem
- 测试结果、截图、命令、响应样例

如果某个判断没有证据支撑，要明确写成：

- 当前判断
- 暂时推测
- 这个阶段的取舍

## 8.1 链接规则

后续在博客、专栏总览、wiki 型文档里写 Markdown 链接时，默认使用真实 GitHub 链接，不使用相对路径。

统一写法：

- `https://github.com/tzrea1-Q/ai-learning-bootcamp/blob/main/<path>`

例如：

- `https://github.com/tzrea1-Q/ai-learning-bootcamp/blob/main/README.md`
- `https://github.com/tzrea1-Q/ai-learning-bootcamp/blob/main/00-docs/01-训练主线/00-学习总览.md`

不要写成：

- `../README.md`
- `./README.md`
- `../00-docs/...`

原因很简单：

- GitHub 真实链接更适合外部平台阅读和跳转
- 脱离本地仓库上下文后，相对路径很容易失效

### 8.2 证据呈现规则

项目型博客不要只给判断，要给读者能看到、能验证、能模仿的材料。

默认至少落 2 类具象证据：

- 1 段源码或接口契约片段
- 1 条响应样例、测试结果或命令输出
- 1 张截图或 1 个素材链接

如果文章写的是工程问题或阶段收口，优先顺序是：

1. 先交代项目背景
2. 再放真实现象
3. 再给代码落点或契约落点
4. 最后再讲判断

图片如果要直接嵌入外部平台，可在 GitHub 图片链接后追加 `?raw=1`。

## 9. 平台直发标准

后续文章默认按“写完即可发”来验收，至少满足下面这些要求：

1. 开头 3 段内进入具体问题
2. 没有大段行业铺垫
3. 没有空泛价值宣言
4. 每一节都有明确功能，不凑数
5. 全文能删掉至少 10% 的地方，必须先删完再定稿
6. 至少有一个明确判断，不做平铺直叙
7. 至少有一个可复用方法，不只讲个人经历
8. 俏皮表达不超过全文的 5%
9. 删掉情绪词后，文章仍然成立
10. 作者本人读完不会产生“这段我还得再改一遍”的冲动
11. 读者只看这一篇，也能明白项目背景和本文焦点
12. 至少出现 2 类具象证据，不只有观点总结

## 10. 落稿前自检

每篇文章写完后，按这个顺序自检：

1. 能不能用一句话说清这篇文章只讲什么
2. 第一屏里有没有废话
3. 哪三段可以直接删掉
4. 哪一句最像套话，把它改掉
5. 哪一句最像在表演，把它收回去
6. 有没有一个判断写得不够硬
7. 有没有一个例子能替代抽象表述
8. 有没有一句俏皮话超过分寸
9. 读者看完能不能立刻复述我的核心观点
10. 如果今天就发，我愿不愿意署名

## 11. 后续执行顺序

以后每次写博客，默认按这个顺序：

1. 先读本文档
2. 再看 `01-templates/blog/技术博文模板.md`
3. 从日报、周报、`docs/traces/`、方案文档、postmortem 里抽题
4. 先写“一句话问题”和“一句话判断”
5. 正文完成后，主动删一轮
6. 只要还觉得“有点虚”，就继续删

## 12. 当前专栏的一句话定义

用真实项目记实账，用工程判断做骨架，用一点点分寸刚好的俏皮，写出能直接发、也真有用的技术文章。
