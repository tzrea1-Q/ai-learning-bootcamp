# 失败复盘｜`POST /rewrite` 风格漂移与轻度扩写

## 1. 事件概述

`week01-text-api` 的 `POST /rewrite` 在最小功能上已经可用，但第一次真实样例回归暴露出一个典型问题：

- 接口能返回结果
- 但返回不一定符合“轻度改写”的预期

具体表现为：

- 短句被改得更书面
- 某些句子会轻度扩写
- 风格容易从“改写”漂移到“说明文重写”

这类问题不会让接口报错，却会直接影响可交付性。

## 2. 影响

- 影响接口：`POST /rewrite`
- 影响判断：如果只看“是否返回了中文结果”，会误判接口已经稳定
- 影响工程质量：prompt、响应清理、回归样例和测试之间没有形成闭环时，风格漂移会反复出现

## 3. 根因分析

### 直接原因

初版 prompt 对输出风格约束太宽，只要求“更清晰、更自然地改写”，没有明确限制：

- 不要加标题
- 不要解释
- 不要明显扩写
- 原文很短时只做轻度润色

### 深层原因

最初更关注“接口能不能通”，没有把“输出是否仍像改写而不是重写”当成显式验收项。

### 为什么没有更早发现

因为一开始缺少真实样例回归，只要模型返回的是中文句子，就容易被当成“已经正常”。

## 4. 处理过程

本轮处理做了 3 件事：

1. 先跑真实样例，而不是只看 mock 测试
2. 收紧 `rewrite` prompt，加入标题、长度和输出形式约束
3. 在结果侧补最小清理，处理“改写后：”前缀和自动标题

## 5. 修正后状态

修正后得到的结论是：

- 显式标题和“改写后：”前缀已经被收住
- 输出长度整体更接近原文
- 仍然存在轻微书面化倾向

这说明问题从“没有约束”推进到了“有约束但仍需继续观察”。

## 7. 教训

- LLM 接口的质量问题，很多不是“报错”，而是“风格漂移”
- 这类问题不能只靠感觉判断，必须有真实样例回归
- 只修 prompt 不够，必要时还要补结果清理和测试
