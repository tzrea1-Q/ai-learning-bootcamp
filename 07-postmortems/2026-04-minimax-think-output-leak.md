# 失败复盘｜MiniMax 返回 `<think>` 片段导致接口输出不稳定

## 1. 事件概述

在 `week01-text-api` 的 MiniMax 连通性验证过程中，接口虽然成功返回了 `200` 和有效正文，但响应内容中混入了 `<think>...</think>` 推理片段。

这意味着“请求成功”不等于“结果可直接对外输出”。如果不做额外处理，前端或调用方会直接收到不稳定、不可控的中间推理内容。

## 2. 时间线

- 2026-04-03：运行 `scripts/test_minimax.py`，确认 MiniMax API Key、Base URL 和请求结构可用。
- 2026-04-03：检查真实返回内容时，发现正文前包含 `<think>...</think>` 片段。
- 2026-04-03：在 `app/main.py` 中补充响应清洗逻辑，统一移除 `<think>` 内容后再返回客户端。

## 3. 影响

- 影响了什么功能：`/summarize`、`/key-points`、`/rewrite` 这类直接透传模型输出的文本处理接口。
- 影响了哪些验证结果：如果只验证 HTTP 状态码和是否“有返回”，会误判为接口已经稳定可交付。

## 4. 根因分析

- 直接原因：MiniMax 当前返回的正文中可能包含模型推理过程，而业务接口默认把 `message.content` 直接作为最终输出。
- 深层原因：接口设计初期只验证了“能否调通”，没有提前定义“什么样的响应才能视为可交付结果”。
- 为什么之前没有发现：最初的验证重点放在鉴权、URL 和请求结构是否正确，没有把“输出清洁度”当成显式验收项。

## 5. 处理过程

- 做了哪些尝试：先运行真实连通性脚本确认现象，再检查返回结构是否稳定出现在 `choices[0].message.content` 中。
- 哪些尝试无效：仅根据 HTTP 200 或 `base_resp.status_code=0` 判断成功，无法发现这类输出污染问题。
- 最终怎么解决：在 `app/main.py` 中增加 `<think>...</think>` 清洗逻辑，并补测试保证接口返回的是清洗后的文本。

## 6. 教训

- 技术层面：LLM 接口的“成功”必须包含输出结构和内容质量，不是只有网络层成功。
- 工程层面：真实 API 验证后，要立刻把异常现象转成代码规则或测试，而不是留在脑子里。
- 流程层面：以后每次新增模型接口，都要把“脏输出、空输出、异常结构”列入验收清单。
