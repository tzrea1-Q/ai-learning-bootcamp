
# 项目概览

## 1. 项目定位

`week01-text-api` 是第 1 周的基准项目，用来完成一个最小但完整的 AI 文本处理 API 闭环。

它不是一个“功能很多”的产品，而是一个训练项目。当前重点是：

- 能启动一个可靠的 FastAPI 服务
- 能通过统一接口接入 MiniMax
- 能提供最小文本处理能力
- 能写出测试、文档和复盘
- 能把模型调用纳入工程流程，而不是停留在脚本试验阶段

一句话概括：

这是一个用 FastAPI 封装 MiniMax 文本处理能力的最小工程化样板。

## 2. 当前能力边界

当前项目已经支持以下能力：

- 健康检查：`GET /health`
- 文本总结：`POST /summarize`
- 要点提取：`POST /key-points`
- 文本改写：`POST /rewrite`
- 真实 MiniMax 连通性验证
- 自动化测试
- 最小日志输出

当前还没有正式支持的内容：

- 统一错误响应格式
- 请求历史持久化
- PostgreSQL 实际接入业务流程
- 用户身份、鉴权、限流
- CI / 自动部署
- 更稳定的输出格式约束

## 3. 为什么这个项目值得保留

虽然它现在规模很小，但它已经把 AI 项目里最关键的几个工程问题连起来了：

- 输入如何约束
- Prompt 如何集中管理
- 上游模型如何封装
- 脏输出如何清洗
- 异常如何转换成 HTTP 错误
- 如何写 mock 测试和真实联调
- 如何记录失败案例

这使它适合作为后续项目的“起始模板”，而不仅仅是一次性练习。

## 5. 各目录职责

### `app/`

放应用主代码。

- `main.py`：FastAPI 路由、请求/响应模型、任务执行逻辑
- `minimax_client.py`：上游 MiniMax 请求封装和最小日志

### `docs/`

放项目级说明文档，帮助你快速理解这个项目的设计边界和维护方式。

### `scripts/`

放辅助脚本。

- `test_minimax.py`：真实调用 MiniMax，验证本地配置和网络链路

### `tests/`

放自动化测试。

- 接口测试
- 客户端测试
- pytest fixture

## 6. 运行前提

要跑通当前项目，你至少需要：

- Python 3.14 左右的本地环境
- 虚拟环境 `.venv`
- 已安装 `requirements.txt`
- 有效的 `MINIMAX_API_KEY`
- 正确的 `MINIMAX_BASE_URL`

如果只是运行测试中的 mock 场景，不一定需要真实 API Key。
如果要跑 `scripts/test_minimax.py` 或真实接口联调，则必须准备有效配置。

## 7. 当前最小交付结果

从训练项目角度看，当前已经形成了这些可交付资产：

- 可运行的 API 服务
- 可调用的文本处理接口
- 接口文档和调试命令
- 自动化测试
- 失败案例复盘
- 日报记录

这些资产比“写过一段调用模型的脚本”更有价值，因为它们可以复用、解释和扩展。
