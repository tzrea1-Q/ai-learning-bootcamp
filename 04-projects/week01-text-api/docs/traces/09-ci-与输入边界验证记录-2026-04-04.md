# CI 与输入边界验证记录｜2026-04-04

## 1. 背景

本次验证对应两项可选收口任务：

1. 为 `week01-text-api` 补一个最小 GitHub Actions 测试工作流草稿
2. 给文本接口补一条明确的输入边界约束，并同步写进代码、测试和文档

本轮目标不是把 CI 做完整，而是先把第 2 周要用的最小骨架提前放进仓库，同时把“输入不能无限长”从隐性假设变成显式契约。

## 2. 本轮改动

### 2.1 GitHub Actions

新增文件：

```text
.github/workflows/week01-text-api-pytest.yml
```

工作流当前能力：

- 在 `push / pull_request / workflow_dispatch` 下触发
- 只关注 `04-projects/week01-text-api/**` 和工作流文件自身
- 在 `04-projects/week01-text-api` 子目录安装依赖并运行 `pytest`

### 2.2 输入边界

当前统一文本输入新增约束：

- `text` 去掉首尾空白后不能为空
- `text` 当前最大长度为 `4000` 个字符
- 超长输入返回 `422`

代码落点：

- `app/main.py`
- `tests/test_text_endpoints.py`
- `README.md`

## 4. 输入边界实际返回验证

### 4.1 验证方式

使用本地 `TestClient` 直接请求：

```python
response = client.post("/summarize", json={"text": "a" * 4001})
```

### 4.2 实际返回

```text
422
{"detail":[{"type":"value_error","loc":["body","text"],"msg":"Value error, text must be at most 4000 characters","input":"<4001 chars omitted>","ctx":{"error":{}}}]}
```

说明：

- 原始返回里包含完整 4001 个字符输入，这里为可读性做了截断展示
- 实际接口确实命中了 `422`
- 错误信息已明确说明当前最大长度为 `4000`

### 4.3 结论

- 输入边界已在接口层真实生效
- 文档里的“最大长度 4000”不是空写

## 5. GitHub Actions 工作流本地检查

### 5.1 检查方式

使用本地 Python + PyYAML 读取 workflow 文件，确认最关键的结构存在：

- `name`
- `on`
- `jobs.pytest`
- `working-directory`

### 5.2 实际输出

```text
Week01 Text API Pytest
['pull_request', 'push', 'workflow_dispatch']
['pytest']
04-projects/week01-text-api
```

### 5.3 额外说明

PyYAML 在某些场景下会把顶层 `on` 解析成布尔值，因此本地检查脚本额外兼容了这个行为。

这不影响 GitHub Actions 自身对 workflow 的解析。
