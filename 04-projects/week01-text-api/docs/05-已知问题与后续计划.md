# 已知问题与后续计划

## 1. 当前已知问题

### 1.1 `rewrite` 输出风格不够稳定

当前 `POST /rewrite` 已经能工作，但真实调用时可能出现：

- 改写过度
- 自动加标题
- 结果比原文长很多
- 偏“重新写一段文章”而不是“按原意改写”

这说明接口已经可用，但还不够稳。

### 1.2 错误响应还不统一

当前项目虽然已经区分了 `422 / 500 / 502`，但错误结构还没有统一。

现在的错误返回主要依赖 FastAPI 默认行为和 `HTTPException(detail=...)`。

这会带来几个问题：

- 前端不容易统一处理
- 错误码和错误信息不够规范
- 后续扩接口时容易风格不一致

### 1.3 日志仍然是最小版本

当前日志已经足够排查最基础问题，但还不适合更复杂的生产场景。

还缺少：

- 请求耗时
- trace id
- 结构化日志
- 更细的业务事件日志

### 1.4 数据库尚未真正接入

虽然仓库里已经有：

- `docker-compose.yml`
- `DATABASE_URL`

但数据库目前还没有参与主链路。

这意味着：

- 没有请求历史
- 没有任务记录
- 没有评测记录

### 1.5 当前还没有提交记录与 CI

从训练角度看，今天的代码和文档进展已经足够明显，但仍缺：

- 一次明确 commit
- CI 留痕

这会影响后续的周报证据链完整性。

## 3. 建议的下一步优先级

### 第一优先级：稳定已有接口

先不要继续堆新功能，优先把现有接口做稳。

建议动作：

1. 收紧 `rewrite` prompt
2. 明确输出长度和风格要求
3. 准备 5 到 10 条真实样例做回归测试

### 第二优先级：统一错误格式

建议定义一套统一错误结构，例如：

```json
{
  "code": "UPSTREAM_REQUEST_FAILED",
  "message": "MiniMax request failed",
  "detail": "..."
}
```

这样后续前端、日志、测试都会更一致。

### 第三优先级：把数据库接进主链路

接入顺序可以先轻后重：

1. 请求历史表
2. 任务记录表
3. 评测记录表

先解决“有记录”，再解决“记录得漂亮”。

### 第四优先级：补 commit 和 CI

当前最小交付已经形成，下一步应该把它固定下来：

1. 整理改动
2. 写一次明确 commit
3. 视情况补 GitHub Actions

## 4. 不建议现在做的事

在当前阶段，不建议立刻做这些：

- 接多个模型供应商
- 过早抽象 provider 层
- 引入复杂中间件体系
- 做多用户和鉴权系统
- 加过重的日志平台

原因很简单：

第 1 周的目标是最小闭环，不是提前把项目做成通用平台。

## 5. 明日最小起步动作

如果明天只做一件事，建议做这个：

1. 选 5 条 `rewrite` 样例
2. 人工跑真实接口
3. 记录哪些输出偏扩写或失真
4. 调整 prompt
5. 固化成测试或文档说明

这是当前最能提升项目质量的最小动作。
