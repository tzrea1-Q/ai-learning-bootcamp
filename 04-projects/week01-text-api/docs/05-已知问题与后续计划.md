# 已知问题与后续计划

## 1. 当前已知问题

### 1.1 `rewrite` 输出风格不够稳定

当前 `POST /rewrite` 已经能工作，但真实调用时可能出现：

- 改写过度
- 自动加标题
- 结果比原文长很多
- 偏“重新写一段文章”而不是“按原意改写”

这说明接口已经可用，但还不够稳。

2026-04-04 已做过第 1 轮回归和修正，详见 `traces/06-rewrite-回归记录-2026-04-04.md`。

当前状态更新为：

- “自动加标题 / 改写后前缀” 已增加提示词约束和结果清理
- “明显扩写” 风险已有所收敛，但仍需要更多真实样例继续观察
- 书面化倾向仍然存在，后续需要进一步通过 prompt 或评测样例收紧

### 1.2 错误响应还不统一

这个问题已在 2026-04-04 完成第 1 轮收口：

- `500 / 502` 已统一为 `code + message + detail + request_id` 结构
- 已补测试覆盖配置错误、请求失败和上游结构异常
- 已补本地验证记录，见 `traces/07-统一错误响应验证记录-2026-04-04.md`
  和 `traces/10-request-id-错误响应验证记录-2026-04-05.md`

当前剩余限制：

- `422` 仍保留 FastAPI 默认校验结构，尚未纳入统一错误层
- 当前只完成了错误响应和异常处理日志的最小 `request_id` 链路，还没有继续下沉到更完整的结构化日志和上游请求链路

关于 `422` 的明确决策：

- 2026-04-05 的决定是：当前阶段继续保留 FastAPI 默认 `422` 结构，不进入统一错误层

原因：

- `422` 属于输入校验层问题，和 `500 / 502` 的服务侧、上游侧错误语义不同
- 现阶段统一 `422` 的收益，低于继续补日志、超时/重试、CI 和 Docker 稳定化
- 如果后续外部调用方对“所有错误统一格式”提出明确要求，再单独开任务处理

### 1.3 日志仍然是最小版本

当前日志已经足够排查最基础问题，但还不适合更复杂的生产场景。

还缺少：

- 请求耗时
- trace id
- 结构化日志
- 更细的业务事件日志

### 1.4 数据库尚未真正接入

虽然仓库里已经有：

- `docker-compose.yml`
- `DATABASE_URL`

但数据库目前还没有参与主链路。

这意味着：

- 没有请求历史
- 没有任务记录
- 没有评测记录

### 1.5 当前还没有提交记录与 CI

从训练角度看，今天的代码和文档进展已经足够明显，但仍缺：

- 一次明确 commit
- CI 留痕

这会影响后续的周报证据链完整性。

## 3. 建议的下一步优先级

### 第一优先级：稳定已有接口

先不要继续堆新功能，优先把现有接口做稳。

建议动作：

1. 收紧 `rewrite` prompt
2. 明确输出长度和风格要求
3. 准备 5 到 10 条真实样例做回归测试

### 第二优先级：继续补强统一错误层

下一步可继续补：

1. 是否把 `422` 也纳入统一格式
2. 是否继续把 `request_id` 下沉到更完整的日志链路
3. 是否让日志与错误码一一对应

其中第 1 条当前结论更新为：

- 短期先不做，保留为第 2 周之后的条件触发项，而不是默认必做项

### 第三优先级：把数据库接进主链路

接入顺序可以先轻后重：

1. 请求历史表
2. 任务记录表
3. 评测记录表

先解决“有记录”，再解决“记录得漂亮”。

### 第四优先级：补 commit 和 CI

当前最小交付已经形成，下一步应该把它固定下来：

1. 整理改动
2. 写一次明确 commit
3. 视情况补 GitHub Actions

## 4. 不建议现在做的事

在当前阶段，不建议立刻做这些：

- 接多个模型供应商
- 过早抽象 provider 层
- 引入复杂中间件体系
- 做多用户和鉴权系统
- 加过重的日志平台

原因很简单：

第 1 周的目标是最小闭环，不是提前把项目做成通用平台。

## 5. 明日最小起步动作

如果明天只做一件事，建议做这个：

1. 选 5 条 `rewrite` 样例
2. 人工跑真实接口
3. 记录哪些输出偏扩写或失真
4. 调整 prompt
5. 固化成测试或文档说明

这是当前最能提升项目质量的最小动作。
