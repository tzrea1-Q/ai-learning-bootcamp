# 接口契约

## 1. 对外接口风格

当前项目对外暴露的是标准的 `HTTP + JSON` REST 风格接口。

接口由 FastAPI 提供，调用方式是：

- `GET /health`
- `POST /summarize`
- `POST /key-points`
- `POST /rewrite`

## 2. 上游模型接口风格

当前项目统一使用兼容 `OpenAI chat/completions` 形态的请求格式，而不是 OpenAI Responses API 格式。默认示例上游是 MiniMax，但接口契约本身不绑定单一供应商。

核心结构如下：

```json
{
  "model": "MiniMax-M2.7",
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "..."}
  ]
}
```

这意味着当前项目更接近“OpenAI Chat Completions 兼容风格”的上游调用方式。

## 3. 通用请求体

三个文本处理接口当前都接收同样的请求结构：

```json
{
  "text": "需要处理的原始文本"
}
```

约束：

- `text` 必填
- `text` 不能是空字符串
- 纯空格会先 `strip()`，再按空输入处理
- `text` 当前最大长度为 `4000` 个字符

空输入或超长输入当前都会返回 `422 Unprocessable Entity`。

## 4. 通用响应体

三个文本处理接口当前都返回统一结构：

```json
{
  "task": "summarize | key-points | rewrite",
  "result": "模型处理后的正文",
  "model": "MiniMax-M2.7"
}
```

字段说明：

- `task`：当前执行的任务类型
- `result`：清洗后的最终输出
- `model`：使用的模型名

## 5. 各接口契约

### 5.1 `GET /health`

用途：

- 最小健康检查
- 验证服务是否已启动

响应：

```json
{
  "status": "ok"
}
```

### 5.2 `POST /summarize`

用途：

- 把一段原文压缩成 2 到 3 句话总结

示例请求：

```json
{
  "text": "FastAPI 是一个现代、高性能的 Python Web 框架，适合快速构建 API。"
}
```

示例响应：

```json
{
  "task": "summarize",
  "result": "FastAPI 是一个现代、高性能的 Python Web 框架，可快速构建 API，并自动生成交互式文档。",
  "model": "MiniMax-M2.7"
}
```

### 5.3 `POST /key-points`

用途：

- 从原文中提取 3 到 5 个关键要点

示例请求：

```json
{
  "text": "FastAPI 支持类型提示、自动文档和高性能，适合快速搭建 Web API。"
}
```

示例响应：

```json
{
  "task": "key-points",
  "result": "- 支持 Python 类型提示\n- 自动生成 API 文档\n- 具备高性能\n- 适合快速搭建 Web API",
  "model": "MiniMax-M2.7"
}
```

### 5.4 `POST /rewrite`

用途：

- 在不改变原意的前提下改写原文

当前输出约束：

- 只返回改写后的最终文本
- 不添加标题、开场白、解释、列表或额外结论
- 尽量保持单段输出
- 长度尽量控制在原文附近，避免明显扩写

示例请求：

```json
{
  "text": "FastAPI 可以更快做 API。"
}
```

示例响应：

```json
{
  "task": "rewrite",
  "result": "FastAPI 可以帮助开发者更高效地构建清晰、可靠的 API 服务。",
  "model": "MiniMax-M2.7"
}
```

## 6. 统一错误响应格式

当前项目对 `500` 和 `502` 已统一使用下面的 JSON 结构：

```json
{
  "code": "SERVER_MISCONFIGURED | UPSTREAM_REQUEST_FAILED | UPSTREAM_INVALID_RESPONSE | ...",
  "message": "稳定、可读的错误摘要",
  "detail": "更具体的定位信息",
  "request_id": "f6386f33-0f8d-4af5-a25d-e4f12a3f7d91"
}
```

字段说明：

- `code`：稳定错误码，方便前端、测试和日志做条件判断
- `message`：面向调用方的错误摘要
- `detail`：更具体的定位信息，通常用于排查
- `request_id`：请求级唯一标识，用于把错误响应、响应头和服务日志做最小链路关联

说明：

- 当前只强制统一 `500` 和 `502`
- 错误响应会同步通过 `X-Request-ID` 响应头返回同一个值
- `422` 仍保留 FastAPI 默认校验结构，避免过早侵入框架原生行为

## 7. 当前错误语义

### `422`

触发条件：

- 输入为空
- 请求体不合法

来源：

- FastAPI / Pydantic 校验层

当前决策：

- 2026-04-05 的结论是：暂不把 `422` 纳入统一错误响应层，继续保留 FastAPI 默认结构

原因：

- `422` 本质上属于框架输入校验层，和当前已经统一的 `500 / 502` 服务侧、上游侧错误不是同一类问题
- 第 1 周当前目标是最小闭环和最小稳定性收口，过早改写 `422` 会扩大改动面和测试面
- 现阶段保留默认 `422` 已足够让调用方判断“请求不合法”，收益暂时低于继续补日志、超时/重试、CI 和 Docker 稳定化

后续触发条件：

- 如果第 2 周之后出现“前端或外部调用方必须统一所有错误结构”的明确需求，再单独立项处理 `422`

### `500`

触发条件：

- 本地配置错误，例如 `UPSTREAM_API_KEY` 缺失
- `UPSTREAM_BASE_URL` 未设置

来源：

- 本地服务侧问题

统一错误响应示例：

```json
{
  "code": "SERVER_MISCONFIGURED",
  "message": "Local service configuration is invalid",
  "detail": "UPSTREAM_API_KEY is not set",
  "request_id": "f6386f33-0f8d-4af5-a25d-e4f12a3f7d91"
}
```

### `502`

触发条件：

- 上游 HTTP 请求失败
- 上游返回结构不符合预期
- 去掉 `<think>` 后没有有效正文

来源：

- 模型供应商或上游通信问题

统一错误响应示例：

```json
{
  "code": "UPSTREAM_REQUEST_FAILED",
  "message": "Upstream request failed",
  "detail": "request timed out",
  "request_id": "3e3c3c68-a398-4d75-95b9-f76a189027de"
}
```

## 8. 当前输出清洗规则

在 `result` 返回给客户端之前，会执行以下处理：

1. 从 `choices[0].message.content` 中提取正文
2. 用正则去除 `<think>...</think>` 片段
3. 对结果做 `strip()`
4. 如果当前任务是 `rewrite`，额外去掉常见的“改写后：/优化后：/润色后：”前缀，以及疑似自动生成的首行标题
5. 如果清洗后为空，则返回 `502`

这个清洗规则是当前接口契约的一部分，而不是临时 patch。

## 9. 当前契约的局限

虽然当前契约已经可用，但还存在这些限制：

- 没有版本号
- `rewrite` 输出风格可能偏扩写，不够稳定
- `.github/workflows/` 里还只是最小 `pytest` 草稿，尚未接入更完整的 CI 检查

这些问题不影响第 1 周最小闭环，但会影响后续扩展和对外稳定性。
