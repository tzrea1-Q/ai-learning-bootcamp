# 开发、测试与调试指南

## 1. 本地启动

### 1.1 初始化环境

```powershell
cd 04-projects/text-api
python -m venv .venv
.venv\Scripts\Activate.ps1
pip install -r requirements.txt
Copy-Item .env.example .env
```

### 1.2 最少需要配置的环境变量

`.env` 中至少要有：

```env
UPSTREAM_API_KEY=your_api_key_here
UPSTREAM_BASE_URL=https://api.minimaxi.com/v1
UPSTREAM_MODEL=MiniMax-M2.7
DATABASE_URL=postgresql://postgres:postgres@localhost:5432/ai_bootcamp
APP_ENV=dev
LOG_LEVEL=INFO
```

说明：

- `DATABASE_URL` 目前主要是为后续接入预留
- `LOG_LEVEL` 当前已影响客户端日志输出

### 1.3 启动服务

```powershell
uvicorn app.main:app --reload
```

启动后可访问：

- `http://127.0.0.1:8000/docs`
- `http://127.0.0.1:8000/health`

## 2. 真实 OpenAI 兼容上游连通性验证

如果你怀疑是 API Key、网络或 Base URL 有问题，不要先查业务代码，先运行最小脚本：

```powershell
.venv\Scripts\python.exe scripts/test_openai_compatible.py
```

这个脚本的目标只有一个：

- 验证上游接口本身能不能打通

如果这个脚本都不通，问题通常不在 FastAPI route，而在：

- `.env` 未生效
- API Key 错误
- Base URL 错误
- 网络不可达

## 3. 自动化测试

运行全部测试：

```powershell
.venv\Scripts\python.exe -m pytest
```

当前测试覆盖包括：

- 健康检查
- 总结接口
- 要点提取接口
- 改写接口
- 空输入校验
- 超长输入校验
- 统一 `500 / 502` 错误响应映射
- OpenAI 兼容客户端成功日志
- OpenAI 兼容客户端失败日志

如果本轮改动涉及真实联调或错误验证，必须同步补一份 `md` 留痕。
当前可参考：

## 4. 推荐开发顺序

如果你要继续扩这个项目，建议按这个顺序：

1. 先改接口或 Prompt
2. 先补 mock 测试
3. 跑 `pytest`
4. 再启动本地服务做真实联调
5. 最后更新 README 或 docs

这样可以避免一上来就把问题归因给上游 API。

## 5. 本地联调命令

服务启动后，可以直接用下面这组命令验证完整流程：

```powershell
Invoke-RestMethod -Method Get -Uri 'http://127.0.0.1:8000/health'

Invoke-RestMethod -Method Post -Uri 'http://127.0.0.1:8000/summarize' `
  -ContentType 'application/json' `
  -Body '{"text":"FastAPI 是一个现代、高性能的 Python Web 框架，适合快速构建 API。"}'

Invoke-RestMethod -Method Post -Uri 'http://127.0.0.1:8000/key-points' `
  -ContentType 'application/json' `
  -Body '{"text":"FastAPI 支持类型提示、自动文档和高性能，适合快速搭建 Web API。"}'

Invoke-RestMethod -Method Post -Uri 'http://127.0.0.1:8000/rewrite' `
  -ContentType 'application/json' `
  -Body '{"text":"FastAPI 可以更快做 API。"}'
```

如果你要验证统一错误响应，可以额外准备两类本地场景：

- 启动前移除 `UPSTREAM_API_KEY`，验证 `500`
- 把 `UPSTREAM_BASE_URL` 指到不可达地址，验证 `502`

## 6. 如何阅读当前日志

当前日志不是完整观测系统，但已经足够排查最常见的问题。

你会看到 3 类日志：

### 请求发出前

用于确认：

- 是否真的发起了上游兼容请求
- 请求打到了哪个 URL
- 当前用了哪个模型

### 请求成功后

用于确认：

- 上游返回状态码
- 是否拿到了响应 ID
- 模型名是否符合预期

### 请求失败时

用于确认：

- 是否是网络超时
- 是否是上游 HTTP 错误
- 是否是配置问题

## 7. 常见开发动作

### 新增一个文本接口

建议按下面顺序：

1. 在 `main.py` 的 `Literal[...]` 中补任务类型
2. 在 `_build_payload()` 中补 prompt
3. 新增 endpoint
4. 在 `tests/test_text_endpoints.py` 增加接口测试
5. 更新 `README.md` 和 `docs/03-接口契约.md`

### 收紧某个接口的输出风格

建议先改：

- `_build_payload()` 中的 prompt

然后做两类验证：

- mock 测试，验证任务语义没有串
- 真实联调，观察输出是否仍扩写或漂移

### 排查“接口能通但结果不对”

优先检查：

1. 上游响应里是否混入 `<think>`
2. prompt 是否过于宽松
3. `result` 是否来自预期字段
4. 是否只是 HTTP 200，但业务输出不可交付

## 8. PostgreSQL 说明

当前项目主流程还没有真正使用 PostgreSQL，但已经预留了容器配置：

```powershell
docker compose up -d postgres
```

当前这个数据库骨架的意义主要是：

- 提前固定环境
- 为后续请求历史、任务记录和评测记录留入口

不要误以为有 `docker-compose.yml` 就代表数据库已经接入业务逻辑。

如果你要验证 Docker 骨架，请至少执行这 3 步：

1. `docker compose config`
2. `docker compose up -d postgres`
3. `docker compose ps`

本仓库已留一次实际验证记录：
