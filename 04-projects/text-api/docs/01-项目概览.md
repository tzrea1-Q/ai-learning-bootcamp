# 项目概览

## 1. 项目定位

`text-api` 是第 1 周的基准项目，用来完成一个最小但完整的 AI 文本处理 API 闭环。

它不是一个“功能很多”的产品，而是一个训练项目。当前重点是：

- 能启动一个可靠的 FastAPI 服务
- 能通过统一接口接入 OpenAI 兼容 Chat Completions 上游
- 能提供最小文本处理能力
- 能写出测试、文档和复盘
- 能把模型调用纳入工程流程，而不是停留在脚本试验阶段

一句话概括：

这是一个用 FastAPI 封装 OpenAI 兼容 Chat Completions 能力的最小工程化样板。

## 2. 当前能力边界

当前项目已经支持以下能力：

- 健康检查：`GET /health`
- 文本总结：`POST /summarize`
- 要点提取：`POST /key-points`
- 文本改写：`POST /rewrite`
- `500 / 502` 统一错误结构
- `X-Request-ID` + `request_id` 最小链路
- `<think>...</think>` 输出清洗
- `rewrite` 的最小风格收口
- 真实 OpenAI 兼容上游连通性验证
- 自动化测试
- 最小日志输出
- PostgreSQL Docker 骨架和验证记录
- `docs/traces/` 与 postmortem 留痕

当前还没有正式支持的内容：

- 请求历史持久化
- PostgreSQL 实际接入业务流程
- 用户身份、鉴权、限流
- GitHub 上实际跑通的 CI / 自动部署
- 对所有错误统一格式的强约束
- 更完整的结构化日志和 trace 链路
- 多供应商适配层和更重的工程抽象

## 3. 文档边界

为了避免多个文件维护同一套信息，当前项目文档职责固定如下：

- 项目根 `README.md`
  - 只做项目入口和导航
- `ENVIRONMENT.md`
  - 环境变量、`.env` 模板、启动方式、验证入口
- `docs/03-接口契约.md`
  - HTTP 接口、错误语义、输入边界、输出清洗规则
- `docs/04-开发测试调试指南.md`
  - 开发流程、测试、联调、日志和排查动作
- `docs/05-已知问题与后续计划.md`
  - 当前限制、明确决策和下一步优先级

## 4. 为什么这个项目值得保留

虽然它现在规模很小，但它已经把 AI 项目里最关键的几个工程问题连起来了：

- 输入如何约束
- Prompt 如何集中管理
- 上游模型如何封装
- 脏输出如何清洗
- 异常如何转换成 HTTP 错误
- 如何写 mock 测试和真实联调
- 如何记录失败案例

这使它适合作为后续项目的“起始模板”，而不仅仅是一次性练习。

## 6. 各目录职责

### `app/`

放应用主代码。

- `main.py`：FastAPI 路由、请求/响应模型、任务执行逻辑
- `openai_compatible_client.py`：上游 OpenAI 兼容 Chat Completions 请求封装和最小日志

### `docs/`

放项目级说明文档，帮助你快速理解这个项目的设计边界和维护方式。

- `docs/traces/`：测试、联调、回归、环境验证等留痕材料

### `scripts/`

放辅助脚本。

- `test_openai_compatible.py`：真实调用 OpenAI 兼容上游，验证本地配置和网络链路

### `tests/`

放自动化测试。

- 接口测试
- 客户端测试
- pytest fixture

## 7. 当前最小交付结果

从训练项目角度看，当前已经形成了这些可交付资产：

- 可运行的 API 服务
- 可调用的文本处理接口
- 明确的环境入口
- 接口文档和调试命令
- 自动化测试
- 验证留痕
- 失败案例复盘
- 日报记录

这些资产比“写过一段调用模型的脚本”更有价值，因为它们可以复用、解释和扩展。
