# 统一错误响应验证记录｜2026-04-04

## 1. 背景

本次验证对应两项收口任务：

- 为 `text-api` 设计并实现统一错误响应格式
- 补齐测试、文档和本地接口验证，确认改动稳定

本轮目标不是扩功能，而是把 `500 / 502` 的错误返回从“能报错”推进到“结构稳定、可测试、可文档化”。

## 2. 本轮实现范围

本轮统一后的错误结构如下：

```json
{
  "code": "稳定错误码",
  "message": "错误摘要",
  "detail": "更具体的定位信息"
}
```

本轮只覆盖：

- `500`：本地服务配置错误
- `502`：上游请求失败或上游返回结构异常

本轮暂不覆盖：

- `422`：仍保留 FastAPI 默认校验结构

## 5. 本地接口联调验证

### 5.1 验证方法

为了避免依赖真实上游波动，本轮本地联调分别启动两次 `uvicorn`：

1. 启动时把 `UPSTREAM_API_KEY` 置空，验证 `500`
2. 启动时把 `UPSTREAM_BASE_URL` 指向 `http://127.0.0.1:9`，验证 `502`

这两轮都先调用 `/health`，确认服务本地启动正常，再调用目标接口验证错误结构。

### 5.2 `500` 场景

启动条件：

- `UPSTREAM_API_KEY=''`
- `UPSTREAM_BASE_URL='https://api.minimaxi.com/v1'`

健康检查返回：

```json
{"status":"ok"}
```

请求：

```http
POST /key-points
Content-Type: application/json

{"text":"需要提炼的信息"}
```

实际返回：

```text
500
{"code":"SERVER_MISCONFIGURED","message":"Local service configuration is invalid","detail":"UPSTREAM_API_KEY is not set"}
```

结论：

- 服务本地可启动
- 接口命中了统一 `500` 结构
- `code / message / detail` 三个字段齐全

### 5.3 `502` 场景

启动条件：

- `UPSTREAM_API_KEY='fake-key'`
- `UPSTREAM_BASE_URL='http://127.0.0.1:9'`

健康检查返回：

```json
{"status":"ok"}
```

请求：

```http
POST /summarize
Content-Type: application/json

{"text":"需要总结的信息"}
```

实际返回：

```text
502
{"code":"UPSTREAM_REQUEST_FAILED","message":"Upstream request failed","detail":"ConnectionError while requesting http://127.0.0.1:9/chat/completions"}
```

结论：

- 服务本地可启动
- 上游不可达时能稳定落到统一 `502` 结构
- `detail` 已收敛成稳定、可读的请求级信息，没有直接把底层系统报错原文暴露给客户端
