# request_id 错误响应验证记录｜2026-04-05

## 1. 背景

本次验证对应 2026-04-05 的收口任务：

- 给统一错误响应补 `request_id` 最小链路
- 至少覆盖 `500 / 502` 两类错误
- 同步补测试、更新 README / 接口契约，并留下新的验证记录

这轮不做大范围日志重构，只交付最小可验收版本：

- 为每个请求生成或复用一个 `request_id`
- 在统一错误响应 JSON 中返回 `request_id`
- 在 `X-Request-ID` 响应头里回传同一个值
- 在统一异常处理日志里打印同一个 `request_id`

## 2. 本轮实现范围

本轮统一后的 `500 / 502` 错误结构如下：

```json
{
  "code": "稳定错误码",
  "message": "错误摘要",
  "detail": "更具体的定位信息",
  "request_id": "请求级唯一标识"
}
```

本轮覆盖：

- `500`：本地配置错误
- `502`：上游请求失败
- `502`：上游返回结构异常

本轮暂不覆盖：

- `422`：仍保留 FastAPI 默认校验结构
- OpenAI 兼容客户端内部日志链路：还没有把 `request_id` 继续下沉到更完整的结构化日志体系

## 5. 本地样例验证

### 5.1 验证方法

为了避免真实上游波动干扰，这轮使用 `TestClient + monkeypatch` 做最小联调：

1. 将 `chat_completion` 替换为抛出 `ValueError`，验证 `500`
2. 将 `chat_completion` 替换为抛出 `RequestException`，验证 `502`
3. 记录状态码、`X-Request-ID` 响应头和 JSON 返回体

### 5.2 `500` 场景

模拟条件：

- `chat_completion` 抛出 `ValueError("UPSTREAM_API_KEY is not set")`

实际输出：

```text
500
500
c2d6a23a-44fc-4a34-9337-c562c4eb47d1
{'code': 'SERVER_MISCONFIGURED', 'message': 'Local service configuration is invalid', 'detail': 'UPSTREAM_API_KEY is not set', 'request_id': 'c2d6a23a-44fc-4a34-9337-c562c4eb47d1'}
```

结论：

- `500` 已返回统一错误结构
- JSON 中存在 `request_id`
- `X-Request-ID` 与 JSON 中的 `request_id` 一致

### 5.3 `502` 场景

模拟条件：

- `chat_completion` 抛出 `RequestException("request timed out")`

实际输出：

```text
502
502
01a1200e-5c70-4678-8d47-c951a9cd54f6
{'code': 'UPSTREAM_REQUEST_FAILED', 'message': 'Upstream request failed', 'detail': 'request timed out', 'request_id': '01a1200e-5c70-4678-8d47-c951a9cd54f6'}
```

结论：

- `502` 已返回统一错误结构
- JSON 中存在 `request_id`
- `X-Request-ID` 与 JSON 中的 `request_id` 一致

### 5.4 异常处理日志样例
