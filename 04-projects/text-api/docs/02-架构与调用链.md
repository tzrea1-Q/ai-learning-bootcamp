# 架构与调用链

## 1. 整体架构

当前项目采用非常轻量的三层结构：

1. HTTP 接口层：FastAPI 路由
2. 任务编排层：Prompt 构造、响应清洗、错误映射
3. 上游客户端层：OpenAI 兼容 Chat Completions 调用和最小日志

这三层目前都在 `app/` 目录中，但职责已经分开：

- `main.py` 负责接口与业务编排
- `openai_compatible_client.py` 负责外部 API 通信

## 2. 请求调用链

以 `POST /summarize` 为例，完整调用链如下：

1. 客户端发送 JSON 请求
2. FastAPI 将请求体解析为 `TextTaskRequest`
3. `validate_text()` 清理空白并校验输入
4. endpoint 调用 `_run_text_task("summarize", request.text)`
5. `_build_payload()` 构造兼容 `chat/completions` 风格的上游请求体
6. `chat_completion()` 发出真实 HTTP 请求
7. 上游兼容接口返回响应 JSON
8. `_extract_result()` 从 `choices[0].message.content` 中提取正文
9. 用正则清理 `<think>...</think>` 片段
10. 返回 `TextTaskResponse`

## 3. 关键设计点

### 3.1 为什么统一用 `TextTaskRequest`

当前三个文本接口都只需要一个字段：`text`。

因此先用统一请求模型有两个好处：

- 降低接口设计复杂度
- 后续可以集中增强输入校验

如果未来 `rewrite` 需要额外参数，比如“风格”“目标长度”，再拆独立请求模型也不晚。

### 3.2 为什么统一用 `TextTaskResponse`

当前返回结构统一为：

- `task`
- `result`
- `model`

这样做的目的是：

- 前端或调用方更容易消费
- 测试更容易编写
- 未来做日志和请求历史记录时，结构更稳定

### 3.3 为什么 Prompt 模板集中放在 `_build_payload()`

Prompt 是业务逻辑的一部分，但不应该散落在每个 endpoint 里。

集中管理后可以：

- 快速看清当前任务如何定义
- 更方便做 Prompt 迭代
- 避免不同 endpoint 出现风格不一致

### 3.4 为什么要清洗 `<think>`

因为真实验证中已经发现：默认示例上游 MiniMax 返回正文时可能混入推理片段。

如果不清洗，会出现这些问题：

- 前端直接展示中间推理内容
- 输出结果不可控
- “请求成功”被误判为“结果可交付”

因此当前项目把 `<think>...</think>` 当作脏输出，在接口层统一移除。

### 3.5 为什么把 `requests` 异常继续往上抛

`openai_compatible_client.py` 只负责调用，不负责替 HTTP 客户端决定最终状态码。

这样分层后：

- 底层负责保留原始异常语义
- FastAPI 层负责把异常翻译成 `500` 或 `502`

这比“在客户端层直接吞掉错误然后返回空字典”更容易排查问题。

## 4. 日志链路

当前日志是最小可用版本，不是完整观测体系。

已记录的日志点：

- 请求发出前
- 请求成功后
- 请求失败时

当前日志内容包括：

- 请求 URL
- 模型名
- 消息数
- 截断后的 prompt 预览
- 响应状态码
- 响应 ID
- 异常堆栈

当前没有记录的内容：

- 请求耗时
- trace id
- 用户 id
- 结构化 JSON 日志
- 持久化日志

## 5. 测试策略

当前采用“两层验证”：

### 第一层：mock 测试

通过 `monkeypatch` 替换 `chat_completion()`，验证：

- endpoint 是否构造了正确 payload
- 接口是否返回统一响应结构
- 空输入是否被拦截
- 上游错误是否正确暴露

### 第二层：真实联调

通过：

- `scripts/test_openai_compatible.py`
- 本地 `uvicorn` + `Invoke-RestMethod`

验证：

- API Key 是否可用
- 网络是否可达
- 模型真实返回是否符合预期

这两层缺一不可。

只有 mock 测试，没有真实联调，会错过上游输出污染问题。
只有真实联调，没有 mock 测试，则很难稳定回归。

## 6. 当前架构的优点与限制

### 优点

- 结构小，容易理解
- 跑通速度快
- 可直接复用为后续模板
- 已具备最小工程味道

### 限制

- 还没有统一错误格式
- 还没有配置对象或 settings 层
- 还没有把数据库接入主流程
- Prompt 与任务类型还是硬编码
- 还没有抽象出 provider 适配层

这个阶段这些限制是可接受的，因为当前目标是第 1 周最小闭环，而不是过早复杂化。
