# 标题

写作前先读：

- `05-blog-posts/00-博客Agent入口.md`
- `05-blog-posts/01-写作规则/项目连载博客写作风格规范.md`

写作硬要求：

- 开头 3 段内进入问题
- 默认删掉第一版里最虚的 10%
- 每段只讲一件事
- 技术范为主，俏皮只点到为止
- 单篇必须自洽，不能默认读者看过前文
- 项目型文章默认至少给出 2 类具象证据：源码/契约、响应/测试、截图/素材
- 项目技术文默认避免“我/你”叙述，优先使用“项目 / 接口 / 当前实现 / 当前阶段”作为主语
- GitHub 文档默认只作补充，不替代正文说明

## 一、项目背景与本文范围

先用最短的篇幅补足背景：

- 这是个什么项目
- 当前已经做到哪一步
- 这篇文章只聚焦哪一个问题
- 如果文章涉及接口或模型调用，先说清上游是谁、对外暴露了哪些能力

如果读者只看这一篇，也应该能在这里跟上。

## 二、本文聚焦的问题

先用一小段说清楚：

- 场景是什么
- 具体问题是什么
- 核心判断是什么

## 三、技术现场

- 问题是怎么暴露出来的
- 如果不处理，会卡住什么
- 如果问题来自 LLM 返回、接口契约或环境链路，把真实现象贴出来

## 四、处理原则与取舍

- 当前阶段先收什么
- 当前阶段不收什么
- 这样取舍的收益和边界是什么

## 五、关键实现

- 具体改了什么
- 为什么这样改
- 哪些事暂时不做

这里默认至少落一个“硬材料”：

- 一段源码
- 一段接口契约
- 一条真实响应
- 一条测试结果
- 一张截图或素材链接

## 六、阶段结果

- 解决了什么
- 还没解决什么
- 下一步最小动作是什么

## 七、经验与实践建议

- 一个判断标准
- 一个排查顺序
- 一个最小检查清单

## 八、素材检查

发稿前快速检查：

- 有没有把项目背景说清楚
- 有没有至少一处代码或契约落点
- 有没有至少一条真实验证结果
- 有没有至少一个截图或素材入口
