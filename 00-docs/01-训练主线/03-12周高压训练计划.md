# 03｜12 周高压训练计划

## 总原则

这 12 周不是自由发挥式学习，而是按周压进度的训练计划。

每周都必须同时留下 4 类结果：

1. 代码或功能进展
2. 文档或说明更新
3. 测试或验证记录
4. 复盘或公开输出素材

## 第 1 周：重建开发者习惯，做最小 LLM API 服务

### 本周目标

- 统一环境
- 起一个 FastAPI 服务
- 接入 MiniMax-M2.7
- 完成 3 个基础文本处理接口
- 建立测试、README、Docker、Git 习惯

### 交付

- `text-api` 项目
- 至少 3 个接口：总结、改写、要点提取
- `README.md`
- `.env.example`
- 5 个以上测试
- Docker 启动方式
- 第 1 份周报

## 第 2 周：把“能跑”提升到“像样”

### 本周目标

- 让第 1 周项目更可靠
- 补日志、重试、超时、错误码、CI

### 交付

- 统一响应格式
- 基础 logging
- 异常处理增强
- GitHub Actions 跑测试
- Docker 稳定化
- 1 份问题复盘

## 第 3 周：做第一个完整小产品

### 可选项目

- 简历优化助手
- 会议纪要整理器
- 客服回复草稿助手
- 长文总结工具

### 交付

- 前后端最小闭环
- 历史记录
- 数据持久化
- 可上线版本
- 1 段演示视频
- 1 份轻量方案文档

## 第 4 周：Prompt、格式约束与效果评测

### 交付

- `20~30` 条测试样例
- 至少 3 组 prompt 对比
- 至少 2 种输出格式策略对比
- 1 份失败案例记录
- 1 篇可公开写作的博客草稿

## 第 5 周：第一个 RAG 系统

### 交付

- 文档导入
- 文档切块
- 检索
- 基于检索结果回答
- 引用片段展示
- 1 份 RAG 方案文档

## 第 6 周：RAG 优化与实验

### 交付

- 至少 30 个问题的 RAG 测试集
- chunk size 对比
- top-k 对比
- 检索拼接策略对比
- 1 份实验结论
- 1 篇实验博客草稿

## 第 7 周：数据库、状态与可观测性

### 交付

- PostgreSQL 正式接入
- 请求记录表
- 任务记录表
- 评测记录表
- 日志增强
- 基础管理接口或查询能力

## 第 8 周：可控的 Vibe Coding / Agent 式编码协作

### 建议新功能

- 历史搜索
- 导出报告
- 用户模板
- 批量处理
- 管理后台雏形

### 交付

- 一项由 Codex 重度参与但可控完成的新功能
- 测试补全
- 风险点记录
- 1 份 AI 协作复盘

## 第 9 周：先做 Workflow，不急着做 Agent

### 可选方向

- 研究资料整理工作流
- 日报生成系统
- 会议材料提炼流程

### 交付

- 清晰的步骤链
- 每步输入输出说明
- 中间状态记录
- 可单步调试
- 1 份工作流方案文档

## 第 10 周：Tool Use 与函数调用

### 至少接入 2 类工具

- 查数据库
- 检索本地文档
- 保存结果
- 调用外部 API

### 交付

- 参数校验
- 失败处理

- 1 份接口或工具使用说明

## 第 11 周：最小 Agent 系统

### 可选方向

- 自动研究助理
- 邮件处理助手
- 任务拆解系统
- 文档理解与行动建议助手

### 交付

- 工具列表
- 中间状态
- 决策路径
- 最终输出
- 失败路径处理

## 第 12 周：收尾、整理、作品集化

### 交付

- 至少 2 个整理完善的项目
- 最终 README
- 架构图
- 测试说明
- 已知问题
- 技术复盘
- 项目 Case Study
- 作品集素材包

## 全程要求

12 周中，每周都必须做到：

- 有 commit
- 有文档
- 有测试
- 有演示
- 有复盘
- 有最小可交付

并且每周都要在这些目录中留痕：

- `03-weekly-reports/`
- `05-blog-posts/`
- `06-solution-docs/`
- `07-postmortems/`
