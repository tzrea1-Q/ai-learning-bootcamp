# 04｜每周交付与验收标准

## 一、为什么必须有验收标准

学习失败的常见原因不是不努力，而是没有明确的通过线。

从这套训练开始，你不允许再用这些模糊表达：

- 差不多做完了
- 基本能跑
- 应该没问题
- 大概懂了

你只能用证据判断：

- 是否能运行
- 是否有测试
- 是否有文档
- 是否能演示
- 是否能解释

## 二、每周最低交付物

### 1. 代码交付

- 有真实功能进展
- 有清晰 commit 历史
- 项目能拉起运行

### 2. 文档交付

- README 至少更新一次
- 周报一篇
- 至少一项输出沉淀

输出沉淀可以是：

- 博客草稿
- 方案文档
- 失败复盘
- 项目案例笔记

### 3. 测试交付

- 核心逻辑至少有基础测试
- 核心接口至少做一次验证
- 记录测试结果
- 测试、联调、回归和样例验证必须至少有 1 份 `md` 留痕文档

### 4. 演示交付

- 一段 `1~3` 分钟 demo 录屏或可访问地址

### 5. 复盘交付

- 写明本周完成、问题、判断、下周计划

## 三、每周通过线

### 通过

满足以下大部分条件：

- 项目可以运行
- 至少完成一个本周核心目标
- 有 README
- 有测试
- 有 commit
- 有演示
- 有周报

### 不通过

任一情况出现即可判定本周不通过：

- 只有想法，没有代码
- 代码不能运行
- 没有 README
- 没有测试
- 没有 commit
- 没有任何演示
- AI 写了很多，但你解释不清关键逻辑

## 四、质量分级

### C 级：勉强完成

- 功能基本跑通
- 工程质量一般
- 文档和测试很薄弱
- 还有明显未解释问题

### B 级：合格

- 功能闭环完整
- 文档齐全
- 测试基础可用
- 错误处理有基本意识
- 可以给别人演示

### A 级：优秀

- 链路清楚
- 测试和日志较完整
- 边界清晰
- README 有质量
- 有失败分析和优化结论
- 有明显工程感

## 五、强制检查表

每周末逐项检查：

- [ ] 本周是否完成了最小可交付
- [ ] 代码是否能在本地跑起来
- [ ] 是否至少做了一次 Docker 或线上验证
- [ ] 是否有测试结果
- [ ] 测试或样例验证是否有可打开的 `md` 留痕文件
- [ ] README 是否更新
- [ ] 是否有周报
- [ ] 是否沉淀了一条博客素材 / 方案 / 复盘
- [ ] 是否有 demo 视频或演示链接
- [ ] 下周目标是否已经明确

## 六、验收时的唯一标准

不是“你有没有花很多时间”，而是：

> 你有没有把结果做出来，并且能解释清楚它为什么这样设计、为什么这样失败、为什么这样修正。
