# 测试与样例留痕规则

## 1. 规则目的

从现在开始，这个仓库里所有测试、联调、回归、评测、样例验证，都不能只停留在终端输出或口头结论。

必须留下可回看、可引用、可复盘的 `Markdown` 记录。

这条规则的目标只有一个：

- 不让“我记得当时测过”这种说法再次出现

## 2. 适用范围

以下动作全部受这条规则约束：

- 跑单元测试
- 跑集成测试
- 手工接口联调
- Prompt / LLM 样例验证
- 回归测试
- Docker / 部署验证
- 错误复现
- 修复前后对比验证

只要发生了“跑测试”或“跑样例”的行为，就必须有对应的 `md` 留痕。

## 3. 强制规则

### 规则 1：没有 `md` 记录，就不算完成验证

以后无论是我还是 AI 助手，只要声称：

- “已经测试通过”
- “已经跑过样例”
- “已经验证”
- “已经回归”

都必须同时给出对应的 `md` 记录文件路径。

如果没有 `md` 文件，这次验证视为未完成。

### 规则 2：记录必须包含实际输入与实际输出

不能只写：

- 测试通过
- 样例正常
- 结果符合预期

至少要写清楚：

1. 测了什么
2. 怎么测的
3. 输入是什么
4. 实际输出是什么
5. 判断结论是什么

### 规则 3：关键输出必须直接写进 `md`

关键结果不能只存在终端历史里。

对于重要验证，必须把以下内容直接放进 `md`：

- 关键命令
- 关键输入样例
- 关键返回结果
- 修复前后差异

如果原始输出过长，可以：

- 在 `md` 里保留关键片段
- 再额外引用 `json / txt / csv` 原始文件

但 `md` 本体不能只留一句“见附件”。

### 规则 4：日报和周报必须引用记录路径

当日如果跑过测试或样例：

- 日报里必须写明对应的 `md` 记录路径

当周如果做过关键验证：

- 周报或验收文档里必须引用对应的 `md` 记录路径

### 规则 5：修复类工作必须优先保留前后对比

如果任务是：

- 修 bug
- 收紧 prompt
- 做回归
- 调整接口行为

那么 `md` 记录里必须优先保留：

- 调整前的实际返回
- 调整后的实际返回
- 对比观察

## 4. 推荐存放位置

优先按下面顺序放：

1. 当前项目的 `docs/`
2. 当前项目下专门的验证记录目录
3. 如果是失败案例或事故复盘，可放 `07-postmortems/`

推荐命名方式：

- `docs/2026-04-04-rewrite-回归记录.md`
- `docs/2026-04-04-api-联调记录.md`
- `docs/2026-04-04-docker-验证记录.md`

如果文档已经存在，就继续追加，不要重新散落成多个无关联文件。

## 5. 最低记录结构

每份测试或样例留痕至少要有这些部分：

1. 背景 / 目的
2. 时间
3. 验证对象
4. 验证方法或命令
5. 输入样例
6. 实际输出
7. 结论
8. 后续动作

建议直接使用：

- `01-templates/project/测试与样例记录模板.md`

## 6. 执行口径

以后在这个仓库里，默认采用下面的判断口径：

- 跑过但没写 `md`：不算完成
- 写了结论但没写输入输出：不算完整
- 只有终端历史没有文档：不算可复盘
- 只有测试名没有实际结果：不算有效留痕

## 7. 对 AI 协作的额外要求

以后 AI 助手在这个仓库里执行测试或样例时，必须默认同时完成两件事：

1. 跑验证
2. 写 `md` 留痕

除非用户明确要求“这次不要留文档”，否则不能跳过。
