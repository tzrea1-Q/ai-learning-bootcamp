# 日报机制使用说明

## 1. 这套机制解决什么问题

很多人学习时最大的问题，不是没有计划，而是：

- 计划和执行脱节
- 当天做了什么没有证据
- 第二天又从零开始
- 学了很多，但沉淀不下来

这套日报机制要求你每天完成一个闭环：

**早上先定计划 -> 白天执行 -> 晚上补全结果 -> 为明天衔接**

## 2. 每天的标准流程

### 早上

1. 到 `02-daily-reports/` 新建当天日报
2. 从 `01-templates/daily/日报模板.md` 复制模板
3. 先填写今日计划、验收标准和风险提醒
4. 如需 AI 协助，使用 `01-templates/daily/日报输入模板.md`

### 白天

按 Must / Should / Could 执行。

注意：

- 先做 Must
- 卡住超过 30 分钟就记录问题
- 有计划变化要在日报里留下痕迹
- 只要当天跑了测试、联调、回归或样例，就必须同步生成或更新 1 份对应的 `md` 验证记录

### 晚上

回到当天日报，补写：

- 实际完成
- 实际产出
- 遇到的问题
- 今天的收获
- 明日衔接

## 3. 为什么计划要放进日报本体

因为日报不是纯总结，而是：

- 早上的执行约束器
- 晚上的结果记录器

只有把计划和结果放在同一个文件里，你后续复盘时才能真正看到：

- 计划是否合理
- 执行是否稳定
- 哪些任务总被拖延
- 哪些阻塞反复出现

## 4. 推荐执行纪律

### 纪律 1：先写计划，再开始学习

不允许先学 1 小时，晚上再倒填计划。

### 纪律 2：计划必须可验收

不要写“学习 FastAPI”“了解 RAG”。

要写成：

- 完成 FastAPI hello world 接口
- 写出 3 个 API 测试样例
- 跑通 Docker PostgreSQL 容器

### 纪律 3：必须记录未完成事项

日报不能只写成绩，必须写：

- 没完成什么
- 为什么没完成
- 明天怎么继续

### 纪律 4：每天至少留下一项可沉淀素材

例如：

- 一个 bug 复盘
- 一个技术认知升级
- 一个博客标题草案
- 一段可公开输出的话

### 纪律 5：测试和样例必须单独留痕

以后不要只在日报里写：

- 已测试
- 已验证
- 样例正常

必须额外保留一份 `md` 记录，至少写清楚：

- 测了什么
- 怎么测的
- 输入是什么
- 实际输出是什么
- 最终判断是什么

如果没有这份 `md`，本次测试或样例验证视为未完成。

## 5. 每周如何利用这些日报

建议你每周末做一次整理：

1. 打开本周所有日报
2. 汇总本周完成、问题、认知升级和可写博客的点
3. 形成：
   - 周报
   - 博客草稿
   - 技术复盘素材

## 6. 最低执行版本

如果你某天真的很忙，也至少完成这 3 件事：

1. 早上写 3 个 Must 任务
2. 晚上记录实际完成
3. 写明天第一件事做什么
