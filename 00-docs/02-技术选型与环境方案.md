## 二、为什么代码助手选择 Codex

对这套训练来说，你需要的不是单纯的“补全几行代码”，而是一个更接近 **工程协作伙伴** 的工具。Codex 更适合以下任务：

但这里有一个硬要求：

> 你可以让 Codex 帮你写代码，但不能让它替你做技术判断。

## 四、为什么开发环境选择 Windows 本地 + WSL2

### 1. 为什么主开发放本地
你目前阶段最重要的是 **开发效率**。本地开发有这些优势：

- 编辑、调试、断点更快
- 和 IDE、代码助手整合最好
- 前后端联调更顺手
- 学习曲线更平稳

### 2. 为什么必须有 WSL2
你的项目最终大概率会运行在 Linux 环境中。WSL2 的价值是：

- 让你在 Windows 上获得接近 Linux 的命令行和文件系统体验
- 提前暴露路径、权限、脚本等兼容性问题
- 让 Docker 与 Linux 环境对齐更自然

### 3. 为什么不建议直接在云服务器上做主开发
如果把 Linux 云服务器当主开发环境，你会遇到：

- 环境维护开销过大
- 本地编辑和远程同步成本上升
- 容易把大量时间消耗在 SSH、权限、依赖、端口、同步问题上
- 训练重点会偏离 AI 应用工程本身

所以最合理的策略是：

> 本地开发，WSL2 保证 Linux 一致性，云端只负责部署和验证。

## 六、固定工作流

### 日常开发流
1. 在 Windows 本地用 IDE 写代码。
2. 用 Codex 辅助小范围实现与修改。
3. 在 WSL2 中运行 Linux 风格命令。
4. 用 Docker 跑 PostgreSQL / API / 依赖服务。
5. 本地先跑测试，再推到 GitHub。

### Docker 一致性流
1. 本地开发可直接跑 Python。
2. 关键服务必须支持 Docker 启动。
3. 在提交前至少用一次 Docker 验证。
4. 所有正式项目最终要能通过容器形式部署。

### 部署流
1. 代码 push 到 GitHub。
2. 在 Linux 云端或托管平台拉起项目。
3. 配置环境变量。
4. 检查日志、接口、数据库连通性。
5. 留下线上地址和演示视频。
